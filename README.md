# 설비관리 대시보드 (구조 공유용)

설계 문서: `../_웹서비스설계/설계.md` — 아키텍처·데이터모델·화면·단계별 계획은 전부 거기에 있음.
이 README는 실행법만.

완전 클라이언트 사이드 앱. 서버 없음, 파일은 브라우저를 떠나지 않음.

## 실행

```bash
npm install
npm run dev       # http://localhost:5173
npm run build     # dist/ 생성, GitHub Pages / Cloudflare Pages에 그대로 배포
npm test          # lib/*.test.ts 단위테스트 + *.test.tsx 컴포넌트 테스트(vitest) — 캔버스·IndexedDB 등은 별도로 Playwright로 검증
```

## GitHub 백업 (2026-07-21 추가)

이 폴더(`_웹서비스`)만 별도 비공개 저장소 `github.com/rnwlgns22-commits/equipment-dashboard`로
`git subtree`를 통해 백업됨 — 상위 `설비` 저장소에 같이 들어있는 `설비통합`(실제 내부
설비 점검이력·MSDS 등)은 원격에 절대 올라가지 않음.

`설비` 저장소 루트의 `.git/hooks/post-commit` 훅이, `_웹서비스`를 건드린 커밋이 생길
때마다 자동으로 백그라운드에서 `git subtree push --prefix=_웹서비스 websvc-backup main`을
실행함(로그: `.git/websvc-backup-push.log`). 이 훅은 `.git/` 안에 있어서 git으로 추적되지
않으므로, 새로 clone하거나 `.git`이 재생성됐다면 아래로 다시 설치할 것:

```bash
cp _웹서비스/scripts/git-hooks/post-commit .git/hooks/post-commit
chmod +x .git/hooks/post-commit
```

`websvc-backup` 원격이 없다면(다른 컴퓨터 등) 다시 등록:
```bash
git remote add websvc-backup https://github.com/rnwlgns22-commits/equipment-dashboard.git
```

### CRUD 완성 (2026-07-20 추가)

Phase 1~4는 "업로드 → 보기" 흐름 위주였고, 등록된 데이터를 화면에서 직접 고치거나
지우는 길이 없었음(설비 추가만 있고 수정/삭제 없음, 이력은 완전 읽기전용, 매핑
연결선은 선택만 되고 아무 동작 없음, 연결 자체를 만들 방법도 없음, 도면도 한 번
올리면 영구히 안 지워짐). 전부 채움:

- **설비**: 상세 페이지에 수정/삭제 버튼. 삭제 시 ①다른 설비의 연결설비 배열에서도
  정리 ②관련 이력은 안 지우고 고아 이력으로 남김 ③레이아웃 매핑 배치도 같이 제거.
  수기 입력 폼(`AddEquipment.tsx`)과 필드셋을 `components/EquipmentFormFields.tsx`로
  공유.
- **이력**: `/history`에 "+ 이력 추가"(수기 등록), 고아 이력 행마다 "설비 지정…"
  드롭다운(그 자리에서 재매칭), 행마다 삭제 버튼. **(2026-07-21 추가)** 설비 상세
  페이지의 "점검·고장 이력" 카드에도 같은 "+ 이력 추가" 폼과 행별 삭제(✕)를 넣어서,
  이력 브라우저로 건너가 설비를 다시 고를 필요 없이 이 설비ID로 바로 등록/삭제 가능.
- **연결설비**: 설비 상세 페이지 "연결 설비" 카드에서 드롭다운으로 새 연결을
  추가하거나(항상 양방향 갱신) 각 항목 옆 ✕로 해제. 매핑 캔버스 `ConnectionPopover`

  도면 위 배치·구역도 같이 정리, 활성 도면이면 남은 도면으로 자동 전환.
- **상세사양(스펙)** (2026-07-21 추가): 그동안 업로드 파이프라인이 채운 값을 보기만
  했는데, 설비 상세 페이지의 "상세 사양"에 "+ 사양 추가"·행별 값 수정(✎)·삭제(✕)
  추가. 기본정보 수정모드와 별개로 항상 편집 가능 — 키 자체를 바꾸는 건 지원 안 함

- **다중선택/일괄삭제** (2026-07-21 추가): 설비 목록·이력 브라우저 둘 다 행/카드마다

- **Phase C (유지보수·분석)** — 완료. 보기 모드 4종(일반/유지보수/히트맵/타임라인) 전부

  슬라이더로 과거 시점 이동, 그 시점 직전 수리 이력 유무로 "그 무렵 이상 있었음"을
  역산 재현(확정 상태 기록 아님, 화면에 명시). 로컬 빌드+브라우저 검증 완료.

이제 설계.md §11 "설비 레이아웃 매핑" 모듈은 Phase A/B/C 전부 완료.

- **저장소를 localStorage → IndexedDB로 교체(2026-07-20, 중요 버그 수정)**: 도면
  이미지(base64)가 몇 장만 쌓여도 localStorage 용량 한도(브라우저마다 다르지만 보통
  5~10MB)를 넘겨서, 그 이후로는 구역을 그리거나 설비를 옮겨도 `QuotaExceededError`가
  콘솔에만 찍히고 아무 안내 없이 저장이 실패하고 있었음("구역 그렸는데 저장이 안 된다"는
  사용자 리포트로 발견). `mappingStore.ts`의 zustand persist storage를 Dexie(IndexedDB)
  기반으로 교체해 해결 — 기존 localStorage(`fms-mapping-v1`)에 남아있던 데이터는 최초
  로드 시 자동으로 IndexedDB(`fms-mapping-db`)로 1회 이전(마이그레이션)됨. Playwright로
  구역 생성→새로고침 후에도 유지 확인, IndexedDB에 실제로 기록되는지 직접 조회해서 확인.
- **설정/데이터의 JSON 백업이 매핑 데이터까지 전체를 포함하도록 확장(2026-07-20)**:
  원래 설비/이력만 내보내던 것을, 레이아웃 매핑(도면·배치·구역·작업오더)까지 한 파일에
  전부 담게 함. 가져오기도 매핑 데이터가 있으면 같이 복원(`mappingStore.loadSnapshot`).
  구버전 백업(매핑 필드 없음)을 가져와도 설비/이력만 정상 복원되고 매핑 부분은 그대로
